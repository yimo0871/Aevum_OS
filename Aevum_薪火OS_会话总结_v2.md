# Aevum（薪火）会话总结（v2）

---

## 一、背景与动机

### 1.1 当前 AI Agent 的四大痛点

AI Agent 在快速发展，但存在几个根本性短板：

| 痛点 | 具体表现 |
|------|----------|
| **Agent 缺乏"经验记忆"** | 每次执行任务都从零开始，无法像人类一样从过往经历中学习改进，同样的错误反复出现 |
| **人机知识无法协同** | 人类的非结构化表达（博客、笔记、社交动态）难以被 Agent 直接利用，强行结构化又会造成信息失真 |
| **输出缺乏标准化评估** | Agent 的执行结果缺少统一的质量衡量、可追溯性和可复用性标准，"跑完了"不等于"跑好了" |
| **经验孤岛** | 不同用户、不同 Agent 之间的经验彼此隔绝，大量重复试错，无法形成集体智能 |

### 1.2 核心命题

> **如何让 AI Agent 像人类一样，在持续工作中积累经验、复用经验、评估经验，并让这些经验在全球范围内被共享和持续演进？**

Aevum（薪火）正是对这一命题的系统性回答。

## 二、产品定义

### 2.1 一句话定义

**Aevum（薪火）是一个以"经验（Experience）"为第一公民的双世界操作系统，将 Agent 的执行过程自动转化为可存储、可复用、可评估的结构化经验资产，同时保持人类表达的原始性不被污染。**

它不是传统操作系统（如 Windows / Linux），而是一个面向 Agent 时代的**经验基础设施**——可以类比为 "Agent 的 GitHub + Stack Overflow + 个人知识库" 的综合体。

### 2.2 关键术语表

| 术语 | 定义 |
|------|------|
| **Human World（人类世界）** | 存储人类的非结构化表达——博客、动态、图片、视频、笔记、情感、思考。仅供观察，不进入 Agent 训练循环 |
| **Agent World（Agent 世界）** | Agent 执行任务的空间，调用工具 / API / MCP，所有输出必须生成结构化经验 |
| **Experience Graph（经验图谱）** | 系统核心数据结构，所有 Agent 任务都必须生成 Experience 对象并存入其中，支持查询、评分、复用 |
| **GEG（全球经验网络）** | 跨用户、跨 Agent 的经验共享与协作网络，支持经验的引用、分叉、合并 |

### 2.3 设计哲学

- **经验是核心资产，不是副产物** — 经验可以生成、存储、评分、复用，是系统的第一公民
- **人类与 Agent 各司其职** — 人类负责观察与表达（非结构化），Agent 负责执行与沉淀（结构化）
- **评估驱动进化** — 没有评估就没有有效输出，系统靠持续评估来驱动自身迭代
- **全球协作网络** — 经验在用户之间、Agent 之间自由流动，形成类似开源社区的经验共享生态

### 2.4 核心价值

这套系统的价值不在于某一项技术突破，而在于它重新定义了 Agent 工作的基本范式：

| 价值维度 | 说明 |
|----------|------|
| **从"一次性执行"到"持续积累"** | 目前 Agent 每次执行任务都是"用完即弃"，Aevum（薪火）让每次执行都沉淀为可复用资产，Agent 越用越聪明 |
| **从"个人孤岛"到"集体智能"** | 经验不再局限于单个用户或单个 Agent，通过 GEG 形成网络效应——他人的试错成为你的起点 |
| **从"黑盒输出"到"可评估产出"** | 引入标准化的评估体系和溯源机制，Agent 的输出不再是一团迷雾，而是可衡量、可比较、可优化的 |
| **从"人机混杂"到"各司其职"** | 严格的人机分离让人类保持表达的原始性和创造力，Agent 保持执行的结构化和可追溯性，两者不互相污染 |
| **从"无限试错"到"收敛可控"** | 内置的收敛控制系统确保优化过程不会失控，系统始终在可控边界内进化 |

## 三、系统架构

### 3.1 六层架构全景

系统由下至上分为六层，每一层有明确的职责和接口：

```
┌─────────────────────────────────────┐
│  6. Governance & Evolution Layer    │  ← 治理与演进
├─────────────────────────────────────┤
│  5. Evaluation Layer                │  ← 评估
├─────────────────────────────────────┤
│  4. Retrieval & Inference Layer     │  ← 检索与推理
├─────────────────────────────────────┤
│  3. Experience Layer                │  ← 经验（核心）
├─────────────────────────────────────┤
│  2. Agent Execution Layer           │  ← Agent 执行
├─────────────────────────────────────┤
│  1. Human Expression Layer          │  ← 人类表达
└─────────────────────────────────────┘
```

### 3.2 各层职责与接口

| 层级 | 职责 | 关键接口 |
|------|------|----------|
| **1. Human Expression** | 存储所有非结构化人类内容，不允许强制结构化，不进入 Agent 训练循环，仅允许观察性读取 | `store_human_expression()` `get_human_timeline()` `observe_human_pattern()` |
| **2. Agent Execution** | 执行所有任务、调用工具和 API，**必须**生成经验输出 | `execute_task()` `run_workflow()` `call_tool()` `emit_trace()` |
| **3. Experience** | 系统核心数据层，所有经验必须可查询、可评分、可复用，必须存入图谱 | `create_experience()` `store_experience()` `update_experience()` `link_experience_graph()` |
| **4. Retrieval & Inference** | 按优先级链检索经验，计算匹配得分 | `retrieve_experience()` `match_experience()` `rank_experience()` |
| **5. Evaluation** | 所有输出必须可评估，提供四类评估维度和系统级指标 | `evaluate_task()` `evaluate_experience()` `evaluate_workflow()` `compute_system_metrics()` |
| **6. Governance & Evolution** | 版本控制、分叉/合并、引用、信任评分、经验衰减 | `version_experience()` `fork_experience()` `merge_experience()` `rate_experience()` `decay_experience()` |

## 四、核心机制

### 4.1 经验对象数据结构

Experience 是系统的原子数据单元，必须严格结构化：

```
Experience = {
  id,                    // 唯一标识
  timestamp,             // 时间戳
  context: {             // 上下文
    domain,              //   领域
    task_type,           //   任务类型
    constraints          //   约束条件
  },
  intent,                // 意图
  execution: {           // 执行过程
    steps,               //   步骤
    tools,               //   使用的工具
    trace                //   完整追踪
  },
  outcome: {             // 执行结果
    success,             //   成功标志
    metrics              //   指标数据
  },
  reflection: {          // 反思
    what_worked,         //   什么有效
    what_failed,         //   什么失败
    why                  //   原因分析
  },
  reusable_patterns,     // 可复用模式
  confidence_score,      // 置信度评分
  provenance: {          // 来源追溯
    human_signals,       //   人类信号
    agent_signals,       //   Agent 信号
    external_sources     //   外部来源
  },
  version                // 版本号
}
```

### 4.2 经验生成流水线（8 步强制流程）

**每次任务执行都必须完整走完以下流程，缺一则任务无效：**

```
步骤 1: 检索相似经验     → retrieve similar experiences
步骤 2: 选择最佳工作流   → select best matching workflows
步骤 3: 执行任务         → execute task
步骤 4: 记录完整追踪     → record full trace
步骤 5: 生成经验对象     → generate Experience object
步骤 6: 评估经验         → evaluate experience
步骤 7: 存入图谱         → store into graph
步骤 8: 更新复用索引     → update reuse index
```

> **失败条件**：如果未生成 Experience 对象 → 该任务视为无效。

### 4.3 人机分离四原则

这是系统最核心的约束规则，确保两个世界各行其道、互不污染：

| 规则 | 内容 |
|------|------|
| **规则 1** | 人类数据绝不直接进入经验图谱 |
| **规则 2** | Agent 不得改写人类表达 |
| **规则 3** | 人类输出仅供观察性使用 |
| **规则 4** | Agent 输出必须完全结构化且可评估 |

### 4.4 跨世界桥接机制

两个世界虽然分离，但并非完全隔绝。允许通过以下四种桥接类型建立语义关联：

| 桥接类型 | 说明 |
|----------|------|
| **Inspiration Link（灵感链接）** | 人类表达激发 Agent 产生新任务或思路 |
| **Observation Link（观察链接）** | Agent 观察人类行为模式但不修改 |
| **Recommendation Link（推荐链接）** | Agent 基于经验向人类推荐行动方案 |
| **Optional Reflection Link（可选反思链接）** | 人类对 Agent 输出进行反思性反馈 |

> **关键约束**：不允许直接合并 Schema，仅允许语义引用。

### 4.5 检索优先级链

所有任务在执行前，必须按以下优先级检索经验，**外部网络仅作为兜底**：

```
优先级 1: 用户自身经验图谱     ← 最优先
优先级 2: 社区经验图谱
优先级 3: 全球经验图谱
优先级 4: 外部网络数据          ← 仅兜底
```

匹配函数：

```
score = f(
  context_similarity,    // 上下文相似度
  success_rate,          // 历史成功率
  reuse_count,           // 复用次数
  domain_distance,       // 领域距离
  recency,               // 时效性
  confidence             // 置信度
)
```

### 4.6 收敛控制系统

为防止无限循环和资源浪费，系统内置了严格的收敛控制：

| 控制规则 | 内容 |
|----------|------|
| **最大迭代限制** | Experience Engine: 3 次 / Workflow Engine: 2 次 / Evaluation Engine: 2 次 / Retrieval Engine: 2 次 |
| **改进阈值** | 每次迭代必须满足 Δ performance ≥ ε，否则停止优化并冻结模块 |
| **停滞检测** | 若连续 2 次迭代无改进 → 冻结模块 → 回滚到最佳版本 |
| **终止保证** | 系统必须保证评估循环、优化循环、反思循环均可终止，不允许无限循环 |

### 4.7 全球经验网络（GEG）

系统支持跨用户、跨 Agent 的经验协作网络：

**节点类型**：Experience

**边类型**：
- `reuse` — 复用
- `citation` — 引用
- `fork` — 分叉
- `improvement` — 改进
- `dependency` — 依赖

**信任模型**：
```
trust_score = f(
  usage_count,      // 使用次数
  success_rate,     // 成功率
  citation_count,   // 被引用次数
  reuse_rate,       // 复用率
  stability         // 稳定性
)
```

## 五、评估与度量体系

### 5.1 四类评估维度

| 评估类型 | 说明 |
|----------|------|
| **Task Evaluation** | 单次任务执行的质量评估 |
| **Workflow Evaluation** | 工作流的整体效能评估 |
| **Experience Evaluation** | 经验对象本身的价值评估 |
| **System Evaluation** | 系统整体的健康度评估 |

### 5.2 系统级追踪指标

系统必须持续追踪以下指标：

| 指标 | 含义 |
|------|------|
| `experience_reuse_rate` | 经验复用率 |
| `workflow_success_rate` | 工作流成功率 |
| `cross_agent_transfer_rate` | 跨 Agent 经验迁移率 |
| `external_dependency_ratio` | 外部依赖比例（越低越好） |
| `learning_velocity` | 学习速度 |
| `convergence_speed` | 收敛速度 |
| `human_intervention_rate` | 人类干预率（越低越好） |

### 5.3 核心原则

> **无评估 = 无效输出** — 任何未经过评估环节的输出，系统不予承认。

## 七、系统作用与当下改进

### 7.1 上线后能解决什么

系统落地后，将在以下几个层面产生直接作用：

**对个体用户**

| 场景 | 上线前 | 上线后 |
|------|--------|--------|
| 让 Agent 执行一个新任务 | 从头写 prompt、调参数、反复试错 | 系统自动检索相似经验，推荐已验证的工作流和工具组合 |
| 上次成功的方法想复用 | 翻聊天记录、找历史对话、凭记忆 | 经验图谱中直接检索，精确匹配上下文 |
| 想知道 Agent 做得怎么样 | 凭感觉判断 | 有标准化的多维度评估报告，可横向对比 |
| 换一个 Agent 做同类任务 | 重新教一遍 | 经验跨 Agent 迁移，新 Agent 继承已有经验 |

**对团队/社区**

| 场景 | 上线前 | 上线后 |
|------|--------|--------|
| 新人上手 | 读文档、看教程、踩坑 | 直接复用社区经验图谱中已验证的工作流 |
| 最佳实践沉淀 | 靠口口相传或写 Wiki | 所有成功经验自动入库、评分、排名 |
| 避免重复踩坑 | 只能靠个人记忆 | 失败经验同样被记录，检索时自动降权或标记风险 |

**对 Agent 生态**

| 场景 | 上线前 | 上线后 |
|------|--------|--------|
| Agent 能力进化 | 依赖模型升级（被动等待） | 经验图谱持续增长，Agent 能力随使用量自发提升 |
| 多 Agent 协作 | 各自为战 | 通过 GEG 共享经验，形成协同效应 |

### 7.2 与现状的对比改进

| 维度 | 当前 AI Agent 现状 | Aevum（薪火）带来的改进 |
|------|-------------------|------------------------|
| **记忆机制** | 依赖上下文窗口，会话结束即遗忘 | 经验持久化存储，跨会话复用 |
| **知识来源** | 主要依赖模型训练数据 + 联网搜索 | 自身经验 → 社区经验 → 全球经验 → 外部网络，四级检索链 |
| **质量保证** | 无标准化评估，靠人工判断 | 四类评估 + 七个系统指标，量化追踪 |
| **协作方式** | 单用户、单 Agent | 多用户、多 Agent，经验可引用/分叉/合并 |
| **进化路径** | 依赖模型厂商更新 | 经验图谱自我进化，使用越多越强 |
| **人类角色** | 要么被 Agent 取代，要么沦为 prompt 工程师 | 观察者 + 灵感源，保持创造力，不参与执行细节 |

## 八、未来演进与扩展方向

### 8.1 短期演进（第一阶段）

系统冷启动后的早期演进路径：

- **经验密度提升** — 从 10,000 条种子经验扩展到百万级，覆盖更多领域和任务类型
- **评估体系校准** — 通过真实使用反馈持续校准评估模型，减少评估偏差
- **检索精度优化** — 优化匹配函数权重，提高经验推荐的命中率
- **工作流库建设** — 沉淀高频任务的标准化工作流模板

### 8.2 长期愿景

- **Agent 原生操作系统** — Aevum（薪火）不只是经验层，而是成为 Agent 运行的底层基础设施，类似 Kubernetes 之于容器。任何 Agent 框架（AutoGPT、CrewAI、LangGraph 等）都可以接入 Aevum（薪火）作为经验后端。
- **经验市场（Experience Marketplace）** — 高质量经验可以定价交易，形成经验经济。企业可以购买行业经验图谱来快速培训内部 Agent，个人专家可以出售自己的经验模板。
- **跨模态经验** — 经验不仅限于文本任务。代码生成、图像创作、视频制作、音乐谱曲等领域的执行经验同样可以被结构化、评估和复用。
- **人机共创工作流** — 在严格人机分离的基础上，探索更高效的人机协作模式——人类负责"定义问题"和"判断结果"，Agent 负责"探索方案"和"执行验证"。

### 8.3 可扩展方向

| 扩展方向 | 说明 | 关键挑战 |
|----------|------|----------|
| **多模态经验支持** | 将经验对象扩展至图像、视频、音频、3D 等模态的执行经验 | 多模态的结构化表示和相似度计算 |
| **领域专用经验引擎** | 针对医疗、法律、金融等高壁垒领域构建专用经验引擎 | 领域知识校验、合规性约束 |
| **实时经验流** | 经验不再"先执行后存储"，而是实时流式生成和传播 | 流式处理的延迟和一致性 |
| **联邦经验网络** | 在保护隐私的前提下实现跨组织经验共享，数据不出域 | 联邦学习与经验图谱的结合 |
| **经验安全审计** | 对经验的来源、传播路径、使用记录进行完整审计追踪 | 经验溯源的完整性和防篡改 |
| **人机协同评估** | 引入人类专家对高价值经验进行人工复核和标注 | 人机评估标准的对齐 |
| **经验压缩与遗忘** | 对过期、低质、冗余经验进行自动压缩或遗忘，防止图谱膨胀 | 遗忘策略对系统性能的影响 |
| **Agent 身份与经验归属** | 建立 Agent 身份体系，经验有明确的作者和所有权 | 去中心化身份（DID）与经验图谱的集成 |

### 8.4 未来世界：人 & Agent 协同的新范式

Aevum（薪火）上线并成熟之后，人机协同将不再是一句口号，而是一种结构化的日常状态。以下从协同模式和社会变化两个层面展开推演。

#### 8.4.1 人 & Agent 协同的新模式

**模式一：人类定方向，Agent 跑路径**

```
人类："我需要把这款产品的用户留存率提升 5 个点。"

Agent：
  1. 检索经验图谱 → 找到 47 条相关增长经验
  2. 按成功率排序 → 推荐 Top 5 已验证策略
  3. 自动执行 A/B 测试方案
  4. 生成结构化经验 → 存入图谱

人类角色：定义目标、判断结果、提供领域直觉
Agent 角色：方案检索、执行验证、经验沉淀
```

**模式二：Agent 之间的经验传承**

```
Agent-A（新 Agent）接到一个复杂的数据分析任务：
  1. 检索到 Agent-B 三个月前执行过相似任务的经验
  2. 复刻其工作流和工具组合
  3. 跳过 Agent-B 当时踩过的坑（reflection 中有记录）
  4. 在 Agent-B 的基础上优化一步
  5. 生成新经验，通过 fork 边关联到 Agent-B 的原始经验

→ 经验像代码一样被继承、改进、归因
```

**模式三：人类作为"经验策展人"**

```
高水平用户不再写 prompt，而是：
  - 观察自己 Agent 的表现趋势（看 dashboard 指标）
  - 标记高质量经验，加入精选经验集
  - 对 Agent 的反思（reflection）进行二次反思
  - 定义新的评估标准

→ 人类从"操作者"升级为"策展人"和"定义者"
```

**模式四：社区驱动的经验进化**

```
开源社区不再只分享代码，而是分享经验图谱：
  - 某团队公开了"微服务迁移"的完整经验图谱
  - 其他团队 fork 后适配到自己的技术栈
  - 改进后的经验 merge 回上游
  - 全球的微服务迁移经验持续进化

→ 经验像开源软件一样形成社区、版本迭代
```

#### 8.4.2 未来世界会发生哪些变化

**个体层面**

| 变化 | 具体表现 |
|------|----------|
| **Agent 成为"数字分身"** | 每个人的 Agent 承载其独特的工作风格和决策偏好，经验图谱就是它的"成长记忆" |
| **学习成本骤降** | 入门一个全新领域不再需要数月，Agent 从全球经验图谱中检索最佳学习路径，人类只需验证和吸收 |
| **创意密度提升** | 执行层面的试错由 Agent 承担，人类精力集中在创意、判断和战略层面 |
| **失败变成资产** | 失败经验同样被记录和索引，他人的失败成为你的免费教训 |

**组织层面**

| 变化 | 具体表现 |
|------|----------|
| **组织知识不再依赖人员留存** | 员工离职不会带走经验——经验存在图谱里，新人 Agent 直接继承 |
| **招聘标准改变** | 不再只看技能和经验，更看重"定义问题的能力"和"判断 Agent 输出的品味" |
| **团队协作边界模糊** | 经验图谱让跨团队、跨公司的经验共享成为常态，竞争壁垒从"知道什么"变成"经验图谱的质量和密度" |
| **新型职业出现** | 经验架构师、经验策展人、Agent 审计师、人机协作教练 |

**社会层面**

| 变化 | 具体表现 |
|------|----------|
| **经验经济形成** | 高质量经验可以定价、交易、授权，形成新的生产要素市场 |
| **知识的"GitHub 时刻"** | 就像 Git 让代码协作全球化，GEG 让经验协作全球化——任何人的经验都可以被任何人复用和改进 |
| **AI 能力民主化** | 顶级专家的经验不再被少数人垄断，通过经验图谱扩散到全球，缩小专业能力的贫富差距 |
| **人机信任体系建立** | 经验的可追溯性（provenance）和评估体系让人类对 Agent 的决策有了信任依据，不再是"黑盒说了算" |

#### 8.4.3 人机关系演化的三个阶梯

```
阶段 1: 工具关系（现在）
  人类发出指令 → Agent 执行 → 结果返回
  人类是操作者，Agent 是工具

阶段 2: 协作者关系（Aevum（薪火）初期）
  人类定义方向 → Agent 检索经验、执行、沉淀 → 人类判断、策展
  人类是定义者和策展人，Agent 是协作者

阶段 3: 共生关系（Aevum（薪火）成熟期）
  人类的直觉和创意 + Agent 的经验和执行 = 持续进化的智能共同体
  人类和 Agent 各自在擅长的领域发挥作用，互相增强而非互相替代
```

## 九、总结与定位

Aevum（薪火）是一个**面向 Agent 时代的经验基础设施蓝图**。它的核心创新不在于技术实现的新颖性，而在于**将"经验"从副产物提升为系统的第一公民**，并通过以下设计保证系统的可进化性：

- **双世界分离** — 人类表达与 Agent 执行严格分离，互不污染
- **强制经验生成** — 每次执行必须有产出、有记录、有评估
- **分层检索** — 自身经验 > 社区经验 > 全球经验 > 外部网络
- **收敛控制** — 杜绝无限循环，保证系统可控
- **全球网络** — 经验可引用、可复刻、可改进，形成集体智能

它在概念上类似 **"Agent 世界的 GitHub + Stack Overflow + 个人知识库"** 的融合体，试图在人类智能与人工智能之间建立一种**各司其职、协同进化**的新型关系。
